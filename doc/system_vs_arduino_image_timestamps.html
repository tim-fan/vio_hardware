System vs Arduino image timestamps¶

The VIO hardware under test/development uses an arduino to provide timestamps for images, based on the basler camera's exposure-active digital output. This is done as it is expected to provide much more accurate timestamps than those provided by the system clock in the basler ros driver.

This notebook attempts to assess whether the arduino timestamps are in fact more accurate, by plotting/comparing with the system timestamps.

Observations¶

Looking closely at the order that timestamps fall, it appears that the timestamp from the arduino is made first, then the image receives a system timestamp about 20ms later. I believe this is because the system timestamp is produced after the image is fully received, so my assumption is that the 20ms represents the transport delay to send the image from camera to host. In comparison, I believe the arduino is capable of sending the timestamp messages very rapidly.

So it appears that the time taken for the camera to send the image to the host is 20ms longer than the time for the arduino to send a timestamp back to the host.

If the arduino is always faster, then the code responsible for matching timestamps with images can assume that a received timestamp message should be associated with the next received image.

Further questions:¶

How accurate are the system timestamps, in terms of jitter / relative accuracy? If the only problem with the system timestamps is one of latency, this can easily be accounted for with a fixed latency calibraiton and correction.

The main reason for the arduino timestamping is that system time is expected to introduce variable timestamping errors on each frame, due to OS process scheduling.

If this is an issue, one would expect the time differences between system timestamps to differ from time differences between arduino stamps. Investigate whether this is the case:

Observations:¶

While system timestamps do appear a little noisier than arduino timestamps, the differences are only on the order of fractions of milliseconds.

Overall, I'm not too clear on what sort of accuracy is required of these timestamps for a VIO application, but I was roughly targeting accuracy to the millisecond. These plots seem to suggest that millisecond timestamp accuracy will be achieved simply by applying a static latency-compensating offset to the system timestamps.

So perhaps the arduino-based timestamping of images based on digital IO interfacing with the camera is not needed? Eitherway, I will proceed with the arduino-based timestamping approach, as it is almost completed, and it should naturally adapt if there are shifts in the arduino-host timestamping offsets (e.g. due to change in network configuration).

I should read up on how much timestamping error should really be expected using system time - I was expecting at least a few ms, in the case where the driver process is out of context at the time that the camera frame arrives. But I don't really know when context switching happens, how long the process will be paused when out of context, etc.

Conclusions¶

Arduino timestamps are received by the system about 20ms before the images.

Considering how to design a node to match arduino stamps with images, it looks like the node could receive an arduino timestamp, then match it with the next incoming image.

You'd probably want a timeout on the matching (e.g. don't match further than 40ms into future).

It seems like any node will need to assume that the system will continue to exhibit the timing characteristics observed here. If this fails to remain the case, it will become hard to match images with timestamps. Further investigation would be required to determine how the observed timing characteristics change with different camera configurations and/or network configurations.

As noted above, it's not quite clear whether the arduino-based timestamping really provides further accuracy beyond what is achievable using a simple staticly defined latency offset, as could be calibrated by a toolbox such as kalibr.

Notes¶

It makes sense that arduinoBagTimes are roughly 20ms ahead of the image header stamps. From the source (https://github.com/magazino/pylon_camera/blob/indigo-devel/src/pylon_camera/pylon_camera_node.cpp#L491) it looks like the image timestamp is generated only after the image has been fully received on the host system. Further, from the camera documentation it sounds like capture frequency is often limited by the time taken to transfer image data to the host. When I attempt to run at 100Hz, the driver limits me to capturing about 30Hz, or every 33ms. This seems to imply that the transfer time is around 30ms. So if the transfer time is around 30ms, and the image header stamp is generated after this transfer, then seeing a image header stamp 20ms after the arduino reference timstamp is not surprising.

If the above reasoning is correct in assuming that transfer delay is the major source of error in the image header stamps, perhaps the accuracy of the timestamping would be significantly improved by recording the time prior to the 'grab' call, rather than after. Or, ideally, perhaps the 'grab' function could determine the time at which the image has been taken and is ready to transfer.

It makes sense that arduinoTimes preceed arduinoHeaderTimes - the header timestamp is taken at the end of the exposure, and the time_ref (arduinoTime) is retroactively set to the halfway point of the exposure. So we'd expect the arduinoTimes to fall half the exposure time before the arduino header time. However, in this case the arduino times are as far as 40ms before the header times, while the exposure time is only 5ms, so it looks like there is another factor determining the delay. The accuracy of the arduino-to-system clock sync is a likely culprit, as the sync is only attempted once in the setup routine, and then is free to start drifting over time from that point.
